# Feature Concept — Tiers, Usage Limits, Model Selection (+ Referral, Admin)

> **СТАТУС: ЧЕРНОВИК, В ПРОЦЕССЕ (2026-07-10).** Захват из голосового потока владельца (2 batch'а). Структуру можно переформовать. Бэклог-идей верхнего уровня — в корневом `FEATURE_CONCEPTS.md`; этот файл — детальная спека конкретной фичи.

## 0. Суть / зачем
Монетизация + контроль стоимости AI. Разделить пользователей на **уровни (tiers)**, ограничить бесплатных по потреблению AI, дать платным выбор модели/провайдера и бо́льшие лимиты. **Философия:** ВСЁ в приложении можно сделать ВРУЧНУЮ; AI лишь чуть помогает и ускоряет. Проповедник спокойно пользуется бесплатно; хочет быстрее — платит. Вся логика уровней/лимитов — **на сервере, не хакабельна**.

## ⚠️ Текущая реальность (заземлено в `frontend/.env.local`, 2026-07-10)
- **Активный дефолт СЕЙЧАС: `gemini-3.1-flash-lite-preview`** (`AI_MODEL_TO_USE=GEMINI`). Уже дешёвая модель — «свапа o1-mini» НЕ требуется (комментарий `openAI.client.ts:44 // o1-mini` УСТАРЕЛ, врёт).
- OpenAI-фолбэк: `OPENAI_GPT_MODEL=gpt-5-mini`; оптимизация: `gpt-5-mini`; транскрибация: `OPENAI_AUDIO_MODEL=gpt-4o-transcribe`; TTS: `gpt-4o-mini-tts` / `tts-1`; Gemini TTS: 2.5/3.1 flash.
- Вывод: главная работа — **каркас уровней/лимитов + рефакторинг выбора модели**, а не смена дефолта. Модель можно подобрать получше отдельно (владелец: «не самый лучший»).

## 1. Уровни — ТРИ разные оси (операционально разведены, чтобы сходились)
Слово «уровень» несёт три РАЗНЫЕ сущности; храним раздельно, доступ ВЫЧИСЛЯЕМ (иначе рассинхрон).
- **РОЛЬ** (кто ты): `superuser` / `admin` / `user`. Управляет ПРАВОМ менять чужие записи (админка). Поле `role` на юзере (см. §5).
- **Оплаченный tier** (за что платишь): `free / 1 / 2 / 3 / 4…`. Биллинг-намерение (пока платёжки нет — ставит админ/дефолт `free`).
- **Promotion** (временный подарок): `{ tier, expiresAt }`. Даёт эффективный уровень ВЫШЕ оплаченного, с автооткатом по дате.
- **Эффективный tier** = `promotion активен ? max(promotion.tier, paid_tier) : paid_tier` — **промоушн только ПОВЫШАЕТ, никогда не понижает** (родня прозе «даёт уровень ВЫШЕ оплаченного» выше; иначе платящий tier2 с ошибочным promo tier1 был бы понижен). Пример: платит tier 2, promotion tier 3 на 2 мес → effective=3 сейчас, =2 после (откат автоматический — истёк, вывод сам вернулся; руками не откатывать).
- **Доступ к фичам · лимиты usage · доступные модели** — НЕ отдельные поля, а **ВЫЧИСЛЯЮТСЯ из эффективного tier** (один источник → рассинхрона нет по построению).
- **Всё определяется ТОЛЬКО на сервере** (по записи в БД; платёжка — позже). UserID — через Firebase.
- **Бутстрап сейчас:** платёжки нет → дефолт `paid_tier=free`; владелец = `superuser` (email в env); админ проставляет tier/promotion вручную.

## 2. Выбор модели и провайдера
- **free:** сервер держит модель + провайдер по умолчанию (дешёвое). Юзер видит назначенную модель либо не видит ничего.
- **paid:** выбирает **провайдера + модель**. Список **курируем вручную**; **максимум ~4 модели на провайдера** (НЕ весь live-каталог OpenRouter — их слишком много). Динамика цен — опционально, но список слугов курируемый.
- **⚠️ Резолвинг — ТОЛЬКО на сервере:** клиент НЕ шлёт модель в запросе (это small hacking — можно подставить платную/чужую). Сервер читает модель юзера из его серверной записи и использует её.
- **Транскрибация и TTS — тоже выбор модели.** Предпочтение владельца: **всё выбирается в settings** (он по себе почти не меняет настройки в моменте). Дефолты из settings применяются к транскрибации/озвучке; per-use override — опционально, вторично.

### 2.1. Провайдеры, ключи, fallback-цепочка
- **Три ключа с разной экономикой:** OpenAI (платный, твой) — транскрибация (звук→текст); Gemini (бесплатный, ~300 сообщений/день на ОДИН ключ = общий котёл на всех) — текст сейчас; OpenRouter (ключ есть, платный) — дешёвые модели.
- **Реестр провайдеров (расширяемый):** модуль выбора должен позволять ДОБАВИТЬ провайдера без переписывания. Текущие (OpenAI, Gemini) сохраняются; OpenRouter добавляется как новый. У провайдера тысячи моделей → куратор показывает ≤4/провайдера.
- **Fallback-ЦЕПОЧКА (маршрутизация с failover) — общий механизм, не только Gemini→OpenRouter:**
  - На КАЖДУЮ способность (TEXT / TRANSCRIPTION / TTS) — УПОРЯДОЧЕННЫЙ список кандидатов `(провайдер, модель)`, дешёвое первым; пробуем по порядку.
  - **Триггеры перехода на следующего:** rate-limit/квота (429, дневной котёл Gemini выбит), 5xx, таймаут, недоступность/auth. **Различать:** retryable (лимит/5xx/таймаут → следующий кандидат) vs терминальные (плохой вход/схема → НЕ фолбэчим, упадёт везде → сразу ошибка).
  - **Цепочка выводится из effective tier + способности:** free-TEXT = `[Gemini free → дешёвый OpenRouter → …]`; paid = `[выбранная юзером → её фолбэки]`.
  - **Терминал (все исчерпаны):** мягкая ошибка юзеру — «AI временно недоступен, попробуйте позже или сделайте вручную» (созвучно философии «всё можно вручную»).
  - **Память лимита (опц.):** «Gemini выбит на сегодня» → не долбить каждый раз, идти дальше до сброса суток.
- **Транскрибация — тоже кандидат на удешевление:** сейчас OpenAI (платно, твой ключ). Проверить, есть ли на OpenRouter STT-модель дешевле `gpt-4o-transcribe` → если да, поставить первой в цепочку транскрибации, OpenAI — фолбэком (см. OPEN QUESTIONS #5).
- **⚠️ Экономика free:** Gemini-лимит (общий) и транскрибация через OpenAI = ТВОИ ресурсы, не per-user. Значит per-user лимиты usage (§3) + перелив на дешёвый OpenRouter = экономическая защита; лимиты и fallback — ОДНА система, не две.

## 3. Лимиты и дисплей usage
- **Транскрибация:** лимит **минут** (месяц; возможно неделя — уточнить).
- **AI usage / токены:** лимит на AI-функции (сортировка, план, инсайты, направления…). Показывать как «usage», а не голые токены.
- **Виджет в settings** (`/settings`, пустая зона справа-сверху в шапке — владелец обвёл на скриншоте): бейдж уровня (free/paid/admin) · текущая модель (free — назначенная/скрыто) · остаток usage · остаток транскрибации · можно ОБЪЕДИНИТЬ.
- **Кнопка (i)** — описывает, какие функции = «AI usage» (= «фичи AI»; это и был «Fetch.ai» из голоса).
- **usage = 0 → AI-кнопки БЛОКИРУЮТСЯ**, при наведении (hover) тултип «недостаточно usage». Ручные действия остаются доступны всегда.

## 4. Security-модель (несущее требование)
- Вся логика free/paid/usage — **на сервере**, не хакабельна.
- Клиент **не передаёт модель** — сервер сам читает из серверной записи.
- Подменил флаг «paid» на UI → сервер не подтверждает → показываются **0/0/0 / дефолты**, платные опции не активны. Все значения (уровень, лимиты, модель) = **источник сервер**.
- **Эскалация против подмены (детерренс, не только тихие нули):** поймали подмену (клиент прислал значение, которое не должен / флипнул флаг) →
  - **1-й раз:** сигнал «состояние обновлено» + сообщение юзеру «система синхронизирована, ваш текущий уровень/модель: X» (мягко — показали реальное с сервера).
  - **Повтор:** серьёзный warning (отбить желание ломать). Считать попытки на юзера.
  - Базовый отпор (сервер игнорит клиентское значение) — СРАЗУ; эскалация-сообщения — можно отдельной фазой позже. (владелец: «на будущее, но предупреждать посерьёзнее».)

## 5. Admin-панель (нужна даже сейчас)
- **Доступ = ОДИН email**, зашитый в серверную env-переменную. Только он имеет доступ к админке. Валидация — на сервере.
- **Роут `/admin`** → авторизация происходит там. Если юзер = admin → в `settings` показать routing-ссылку «go to admin».
- **Конфиг Default free usage** — выставлять дефолтный usage для free-уровня (иногда 100, 500, иногда 0 единиц).
- Проставлять уровни юзерам; выдавать «N месяцев бесплатно» (напр. при непонимании с юзером — «пользуйтесь 2 месяца бесплатно»).
- Поддержка пользователей. Когда промо закончится — владелец ставит себе admin, остальные — по реальным уровням.
- **Фундамент ролей на вырост (закладываем сейчас):** доступ = один email в env (superuser) — НО поле `role` на юзере хранится сразу, чтобы завтра добавить 2-3 админов / «ответственного за платежи» (выдаёт promotion как компенсацию за сбой) без переписывания.

## 6. Referral / рост (будущая идея)
- Email существующим юзерам об обновлениях.
- Через ~неделю: «пригласи друга → месяц paid» через **реферальную ссылку**, связывающую пригласившего↔приглашённого (email пригласившего / реф-ссылка на регистрации).

## 7. PREP-РАБОТА — рефакторинг ПЕРЕД фичей (делать ПЕРВЫМ)
- **Единый модуль выбора провайдера/модели = реестр провайдеров + политика fallback.** Сейчас раскидано (`openAI.client.ts:43-61` + `structuredOutput.ts:23-42,135` дубль + tts/optimization/transitions). Свести в ОДИН модуль, который: (а) держит РЕЕСТР провайдеров (добавить нового без переписывания; текущие OpenAI/Gemini сохранены, OpenRouter — новый); (б) резолвит модель per-user на сервере; (в) умеет FALLBACK (Gemini лимит → OpenRouter).
- Принципы: **Clean Code · YAGNI · Single Responsibility.** Причесать → красиво, структурно, расширяемо.
- **Аудит текущей paid/premium-реализации** (`models.ts:386-401 isPremium` — мёртвая заглушка): возможно что-то делали, но не по best practice → привести к best practice ИЛИ снести.
- Спросить **Hamming** (оптимальность): текущий выбор моделей/архитектура — не локально ли лучшее; есть ли лучше.

## 8. Cross-cutting (учесть в плане)
- **Синхронизация usage IndexedDB ↔ сервер** (cache coherence): не должно быть, что в IndexedDB закешировано «есть usage», а на сервере уже 0. Источник истины — сервер.
- **Класс C (проектный долг, НЕ эта фича) — вынесено в корневой `FEATURE_CONCEPTS.md`:** чистка env-переменных · убрать устаревшие комментарии (`// o1-mini`) + гигиена по ходу · валидация промптов под разные модели.
- **BYO-key — ОТЛОЖЕНО:** хранение чужих ключей = риск утечки. Пока только ключи владельца (он отвечает за свои). BYO — позже, если решим.

## 9. План реализации — фазы (поток; Ohno + Beck, 2026-07-11)
Порядок владельца (рефакторинг → админка → остальное) + вставленный ТОНКИЙ фундамент данных ПЕРЕД админкой (админке нужно КУДА писать). Каждая фаза — проверяемый вертикальный срез (jidoka: сломалась — стоп, не тащи брак дальше).

- **Фаза 0 — Рефакторинг** (B0): единый модуль = реестр провайдеров (расширяемый) + fallback-политика + аудит мёртвой `isPremium`. ✔ дефолт Gemini 3.1 работает после сборки.
- **Фаза 1 — Фундамент данных** (тонкий): поля на сервере `role` · `paid_tier` · `promotion{tier,expiresAt}` · счётчики usage; сервер читает, вычисляет effective tier (дефолт free/user). ✔ читаю запись юзера → вижу поля + effective.
- **Фаза 2 — Админка** (B5): `/admin`, доступ = email в env (superuser); выставить юзеру role/tier/promotion/default-usage. ✔ зашёл админом → поставил себе paid → отразилось.
- **Фаза 3 — Резолвинг модели** (B2): сервер берёт модель из записи юзера (через модуль Фазы 0) + fallback; клиент НЕ шлёт. ✔ лог — paid→его модель, free→дефолт, Gemini-лимит→OpenRouter.
- **Фаза 4 — UI настроек** (B4): бейдж уровня · селектор (≤4/провайдер) · виджет usage · (i) · usage=0 блок AI-кнопок. ✔ вижу статус/остаток; на 0 — кнопки серые.
- **Фаза 5 — Лимиты usage** (B6): счётчики (транскрибация минуты + AI usage) + enforcement + sync IndexedDB↔сервер. ✔ потратил → счётчик упал; 0 → блок.
- **Фаза 6 — Referral** (B7): реф-ссылки, «пригласи → месяц paid».
- **Фаза 7 — Validate + deploy** (B8): Popper-слом + Gawande на необратимом.

## Валидация по фазам — модальность теста ПОД характер фазы (2026-07-12)
Принцип: каждая фаза закрывается СВОИМ типом проверки (где-то unit, где-то e2e, где-то manual-browser), + дисциплина Ф0 (Codex-сборка → независимый Popper на порцию → гейт), соразмерно сложности.
- **Ф0 Рефакторинг** — unit + полный `test:fast` (бэкенд, UI нет). ✅ 3412/3412.
- **Ф1 Фундамент данных** — unit (effectiveTier: paid/promo/expiry/дефолт) + integration (admin-SDK читает user-doc; tier-поля НЕ client-writable).
- **Ф2 Админка** — integration (email-гейт) + **manual browser** (зашёл админом → выставил tier/promo → отразилось).
- **Ф3 Резолвинг+fallback** — unit (runner по диспозициям) + **e2e** (лог: paid→его модель, free→дефолт, лимит провайдера→fallback).
- **Ф4 UI settings** — **manual browser** (feel: бейдж, селектор, usage-виджет, usage=0 блок+тултип) + unit (компоненты).
- **Ф5 Лимиты usage** — unit + **e2e** (потратил→счётчик↓→0 блок) + тест sync IndexedDB↔сервер.
- **Ф6 Referral** — **e2e** (реф-ссылка→атрибуция→месяц paid) + manual.
- **Ф7 Deploy** — `test:fast`+tsc + manual smoke на прод + Popper/Gawande.
